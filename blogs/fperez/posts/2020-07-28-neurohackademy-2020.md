### LIGO and the Gravitational Wave Discovery

So here's a lovely example of some of the impact of Binder. Back in 2017, you may recall that the Physics Nobel Prize was awarded for the discovery of gravitational waves. This was a prediction made by Einstein back in 1916. It was a theoretical prediction that if a massive gravitational event happened in the universe, like say the explosion of a supernova or the collapse of two black holes into one another, these gravitational events would radiate energy, and that energy would travel in the form of a gravitational wave, which is a stretching and squeezing of space-time itself, and that would travel at the speed of light.

So it's a phenomenon of a wave that travels not unlike light, which is an electromagnetic wave that travels through space and time, but while an electromagnetic wave is a propagation of electric and magnetic fields through space and time, this is a propagation of a distortion of space and time itself. Turns out that even though the prediction was made in 1916, the observation of these took about 100 years. It's extremely hard to make. It was a feat of science and engineering to create a detector that would be able to sense these.

That detector is called LIGO, and it's actually a set of two instruments, one in eastern Washington state and one in Louisiana, that have to solve a very, very tricky measurement problem, which is to detect a motion of the size of roughly one thousandth of the size of a proton relative to a distance of about four kilometers. So what these instruments are are interferometers that have a laser in the corner. That laser is split into two, and it shines back and bounces back and forth down these two arms that are four kilometers each. And if space and time itself stretches, the length of these arms will change differentially. And that difference can be detected because of the interference pattern of the laser light coming back. You need two of them because just about anything will cause ripples and waves into one of them. Somebody walking down the street, a truck going by miles away, somebody dropping a hammer at their home in the next county over, will probably cause ripples that are bigger than the signal you're trying to detect.

### Repo2Docker and Binder Internals

So for those of you who are interested in the internals, as I mentioned, it's kind of a composition of layers. Repo2Docker is the tool that will give you a deterministic build of a Docker container out of a GitHub repository. So the idea is that if you have a repo, it will make a Docker image. It uses and recognizes existing ways of declaring your environment. It supports many languages and it's a fairly lightweight, simple command line tool. You install it, you pip install it, you run Jupyter Repo2Docker and you give it the path, either the URL of a public repo or a local Git repo sitting on your machine. And if it has dependencies explicitly declared, it'll bundle them and give you a Docker container that has all those dependencies prepackaged for you.

What it's doing is cloning the repo, detecting what are the things that are required in a variety of formats, setting up the necessary Docker file, and assembling out of the environment. It'll run your things. It actually gives you tools to fine tune how the build happens and whatnot. It's not limited to running kind of a bare environment that says I need these libraries. You can actually do pretty much whatever you want in the build. It'll build it, it'll push it to the cloud. It does all that for you.

It recognizes dependencies not just in Conda format. Binder actually recognizes requirements in a variety of ways, which means you can do pretty much anything you want, including writing the Docker file yourself. You can sort of say, I don't want any of the automation. I need to do something complicated enough that I need to write the Docker file myself. And you can use it to build your own environments with arbitrary Docker files. You can run your own UI. The interface doesn't have to be Jupyter Notebooks. It could be JupyterLab, it could be RStudio, it could be something else.

And because it is a Docker container, you can run that for testing on basically anywhere that runs Docker. So it could be locally or in the cloud. But it will, if you want it to, it will put it on a JupyterHub that is deployed interactively for you in the cloud. And it will actually run that in a kind of scalable, generic, orchestrated way using a tool called Kubernetes, which is a tool for basically running Docker containers at scale in the cloud. Now, Kubernetes is its own kind of layer of complex engineering. I'm not an expert in Kubernetes. I've never written a single Kubernetes recipe or file or a bit of code. But I don't have to because that layer of deployment is not a layer that I need to right now deal with. And so I build my containers, I tweak them, sometimes I do custom things to share what I need, and Binder does the rest and does the orchestration for me.

### PEP 622: Binder Enabling Python Language Design Collaboration

So here's a little recent fun example of an interaction that I had with the core Python team. So Guido van Rossum, who's the original author of Python had posted a new proposed idea for the Python language for version 3.10 of Python called PEP 622 which adds a rather complex syntactic extension to the language that was a little bit tricky to wrap your head around. And so what I did was I went to the repo that Guido had which is called patma in his GitHub repo, and I said why don't we add Binder support to this repo that implements the idea because it's called structural pattern matching.

And my argument was, well this idea is kind of complex to wrap your head around, and it would be more productive to have a discussion if people could play with it. But it turns out that to play with it you have to build a very specific version of Python 3.10 alpha. To give you a sense, Python 3.9 hadn't been released, so you have to go and fetch the source repo for Python 3.10 and you have to fetch a very specific branch which is not even in the CPython organization but somebody else, one of his colleagues, Brian Bucher, is the one who has the implementation. So you have to go grab the exact branch of 3.10, build it, make sure that you get the build right, and then you can try to play with these ideas.

And I think the focus of Guido and Brandon and others right now is not on the build process. What they're interested in is feedback around whether the idea works or not. So I said why don't we do this. I got it started, then folks like Tim Head, who's one of the leads on Binder, and Chris Holdgraf, who is a neuroscientist who works at UC Berkeley and is also one of the leads on the Binder project, and Matthias Bussonnier, who I mentioned created originally the notebook viewer, they all jumped in to help me because there were weird issues. This was not a trivial Binder to build because it required a custom build of Python itself, but the JupyterHub runs on Python and it doesn't run on Python 3.10. So we had to add a custom version of Python without breaking the running of the Binder itself.

They very kindly helped me over the span of like two days and we were able to provide a playground, the PEP 622 playground, which is basically a simple notebook that is guaranteed to run, it checks that it's running on 3.10, it has a simple toy example, and Guido posted on the Python developers mailing list the links to this so that people could now have hopefully a more informed discussion. I wanted to show this one partly because I had a lot of fun doing this a few weeks ago, but also to illustrate that reproducibility enables collaboration. Making something like this easily reproducible with one click makes it easy for a community to collaborate, in this case on a complex technical discussion around a new feature for the Python language, rather than in the abstract.

### Running Binder at Scale

Now just to wrap up, it turns out that running a service is complicated. Binder is a complex and rapidly growing service. Today Binder has on the order of 150,000 weekly sessions. They cost about $0.02, basically about two cents per session. So this is not a trivial exercise to run this in the cloud. Today Google Cloud, OVH which is a French cloud provider, GESIS Notebooks and the Turing Institute of the UK all provide basically clusters. And when you go to Binder, you don't know where you get a notebook because what you will get is one of the containers in one of these nodes of the federation.

But the point is we're trying to work with a community-driven federated model rather than making this a single point of access that say a company controls. I think this matters because as perhaps a number of you are aware of, there are big battles right now between the publishing industry and the scientific communities.
