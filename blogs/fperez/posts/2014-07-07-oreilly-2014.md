### The Origins of IPython

**Interviewer:** You created IPython back in 2000, is that right? 2001?

**Fernando Perez:** 2001.

**Interviewer:** What problems were you trying to solve at that point?

**Fernando Perez:** Basically, I was trying to switch. I was a graduate student in physics, finishing my PhD, and I was trying to switch away from using proprietary tools to using open tools, and I was trying to simplify a lot of my workflow from many, many programming languages to using less. And when I discovered Python, I realized that I could replace a lot of different tools with just one language that would make it easier to keep kind of a smoother workflow.

But what was missing in Python was a good interactive computing environment. Python has a basic interactive shell, but it doesn't fit the kinds of workflows that you need in scientific computing where you want to run a script, you want to look at data, you want to plot some variables, you want to change the code a little bit and keep a very exploratory workflow. And so that, and being a grad student looking for an excuse not to work on his dissertation and instead do something more fun, was the perfect combination to start writing IPython.

### IPython's Surprising Growth

**Interviewer:** So what surprised you about IPython's evolution?

**Fernando Perez:** I guess the fact that precisely something that began as such a simple personal fix for a problem in my own workflow has grown into such a large project. ### How Much Programming Should Scientists Know?

**Interviewer:** So expanding the scope of this a little bit, how much programming do you feel that scientists need to know?

**Fernando Perez:** It varies quite a bit depending on exactly what we're doing. I think it's important to understand that for scientists, programming has a very different flavor than it does for software developers and software engineers, professional software engineers, in that in scientific contexts, by and large — and this is not universal, but by and large — programming is inherently exploratory and iterative, and it's really about answering a question. Programming really is a tool about answering a question.

There's a lot of ad hoc code that gets written in science, and that's sometimes okay because it may be that that code that you've written only applies to that particular data set. So there is no purpose in spending a month designing a massive framework around that to abstract it seven layers deep. It's okay to write one script that mines that data set and extracts the answers that you need.

But at the same time, scientists also need to learn how to gradually understand when they do need to abstract a little bit, when that data pattern in that data set is going to be similar in other problems, and they should not copy-paste that script 50 times. But they should begin moving it into a few functions and having a little library that they use and unit test that library so they can trust it.

And so finding a way to operate in that continuum that goes from completely exploratory ad hoc work with code or data or an idea or something you read in a paper all the way to building production tools that are full libraries and that are unit tested and that have regression testing machinery, etc., and that creep into the realm of production software engineering — I think that's one of the more interesting challenges because it's a different flavor of programming that I think is different from what the industry has traditionally seen. And I've had really interesting conversations about this with folks like the Microsoft Visual Studio team who come from the heavy-duty, professional, high-end software engineering tool world, and understanding how there's a change in perspective about this has been, I think, valuable.

### On "Everyone Should Learn to Code"

**Interviewer:** Somewhat related to this, what's your take on the notion that everyone should learn to code?

**Fernando Perez:** There's the obvious, I guess, answer to that — that yes, as the world becomes more computational, it's useful for many people to know how to code. I don't know if everyone, but certainly for a lot of people. But I'd like to take a slight twist on the question, which is to ask, well, why do we code?

We code because we are trying to communicate to a machine that doesn't have human-style natural intelligence how to accomplish a task. And in addition to the fact that that means it's a dumb machine, there's also something interesting, I think, in looking at it in this form, which is that we code because in coding we have to clarify, to the point that a dumb machine can understand, our thought process and what it is that we are going to do.

And it means we have to organize the process in a very precise and unambiguous way. We have to perhaps abstract pieces of it so that it can be done by reusing the reusable components, et cetera. And that process of logical, systematic, organized, abstract computational thought — that, I think, is a useful, valuable skill in many contexts in life. Being able to think through a problem in a systematic, organized, logical way, understand what parts of the problem are abstractable and generalizable, understand exactly how to accomplish a process that will complete, is a useful skill.

### Computational Thinking

And so I think there's the act of coding — there's the programming, the mechanics of programming and syntax, et cetera — and there's the aspects that are about what I would call computational thinking.
