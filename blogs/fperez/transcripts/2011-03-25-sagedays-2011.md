### The REPL as a Messaging Protocol

But what I want to talk about now is the idea that if you look at an interactive environment in its most basic form, you'll often see the acronym — it's a REPL, it's a Read-Eval-Print Loop, which basically means take user input, execute it, and provide output. And if you add support for data transfer, it turns out that interactive work and parallel computing actually look very, very similar, because they're basically both things that are trying to manage a namespace — they're basically trying to give instructions to have something happen in a namespace and have something change in that namespace.

If you read MPI code, you can think of it basically as code that is telling — over there in that namespace I want this to happen, or I want you to get this piece of information, or I want that piece of information which you have to be sent back to me. And so, first of all, these steps can happen in multiple processes. This is not news to you guys — I use the Sage notebook all the time. In the Sage notebook, you read in a web browser, it evaluates in a kernel process over there, and you get back your information into the web browser. So this is not news to you guys.

And so what we're trying to do with IPython these days is to spend a fair amount of time carefully sort of looking at what this entailed and what kind of workflows we could achieve if we took some time to sort of look at the patterns that are here and what kinds of things can you do with it. And we realized if you have some place where code is being executed, and you define messages to go back and forth between the thing that is controlling it, but also you specify what information that kernel may put out into other channels that are publicly broadcast channels, so that not only are you sending commands and data back and forth, but you also have a way of the kernel informing others, then there's a number of things that you can do that are interesting.

But that meant that we thought very carefully about what information had to go back and forth, and we defined a messaging protocol between clients and a kernel. We specified it completely — there's a document in our docs that specifies what parts of the communication are meant for direct communication, like execute code or introspect objects, like the things you do with the tab key, the things you do when you do question mark, information about what has been done before — what kinds of things should be broadcast. And once you start thinking about it, you realize that you basically want to separate the two.

### ZeroMQ's Socket Abstractions

And so the beauty of ZeroMQ is that the sockets it creates encapsulate messaging patterns. You're not pushing bytes. You're not pumping the socket for bytes. You say, I need a socket that encapsulates back-and-forth two-way communication. That's a type of socket. And then on that socket you send and receive and you're done. You just send or receive and you're done. Or you say, I have a guy who wants to broadcast information and I have guys who want to listen to the information that is being broadcast. Then you do that. You create a pub socket and that socket broadcasts. And then you create clients that have sub sockets that bind to that socket and then they listen and they can actually filter the streams of information coming from it.

So the library, the abstractions of the library — it seems fairly low level. But it's, in my mind, at the right level. Yes, it is low level. It's not big web frameworks. It's sockets. But they're sockets that represent flows of information, not transfers of bytes. And that actually makes a big difference.

So what we did was, once we understood that, then with the messaging protocol we implemented the interactive part of IPython based on this, where now we have a kernel. The kernel has several sockets. It has a socket that is a multiplexed reply socket where it can take requests from clients, but it can take requests from more than one at the same time, because ZeroMQ makes that absolutely trivial — it's just a different type of socket, so that more than one client can simultaneously talk to the kernel. It has a socket called pub where it broadcasts information that they can all subscribe to. And it can have a request socket, because if the kernel code needs input from its side — say the kernel does raw input — then who's going to give it input? You basically can teleport to the keyboard by having one of the clients have a socket which is taking input where there is a real keyboard and feeding it back there.

So basically we have the architecture for breaking up what is normally a single-process idea of the read-eval loop into two processes, with the possibility for collaboration, with the ability for all of these connections to be severed at any point in time and reappear, for clients implemented in different languages. It doesn't matter what the language of the client is — it can be JavaScript, it can be Java, it can be whatever you want it to be. This is a messaging protocol. We're sending JSON over ZeroMQ. That's all we're doing.

And by the way, the Python bindings for ZeroMQ were written by Brian and now also maintained by Min. So we have end-to-end control. We have Python 3 bindings. You run Cython with no options once and it runs it on every Python. No 2-to-3 here — it generates code that works with either one using hash defines and stuff. Whereas Twisted on Python 3? Yeah, maybe in 10 years.

### Python's GIL and Parallel Solutions

In Python, we have serious problems because the infamous Global Interpreter Lock prevents more than one Python — Python has very good thread support, but threads in Python can't do anything CPU-intensive because there is a lock that prevents more than one Python instruction from running at the same time. You can spawn a bunch of threads to wait for network data, for I/O. You can't do CPU-intensive work that touches any Python-visible structures.

If anybody is interested in this, David Beazley, who used to be at UChicago, has a phenomenal amount of work done on analyzing the internals of the GIL and doing as much as can be done to improve the situation within the GIL. If you want parallelism in-process, there are some things you can do. You can do data parallelism with threaded libraries. You can use a threaded ATLAS. There are tools to do parallel NumPy expressions — numexpr is something that takes NumPy expressions and effectively compiles them, it's sort of a mini virtual machine specialized for NumPy.

Theano is a very interesting project from U Toronto's CS department. That is basically the marriage between a compiler and a symbolic computing system, a computer algebra system. It kind of goes to the question about the marriage of these things. Theano basically takes high-level descriptions of functions in terms of high-level primitives and does symbolic transformations on them to emit fast code, either for multi-core or for GPUs. Very interesting.

If you want to work out of process, Python has had a very good solution — multiprocessing. It's pretty nice. In 3.2, there's something called Python Futures. There's a bunch of other tools. We'd like to think that with IPython, we have something to offer now as well that is actually fairly interesting.

### IPython Parallel: From Twisted to ZeroMQ

As I mentioned before, we've had an implementation using Twisted for parallel computing. But what we did was we took the same diagram that I just explained and the same sockets for interactive work and pushed them into the idea of parallel computing, where now you can have N engines that do the kind of code execution that I talked about. These can be many. These engines all connect to a controller object that looks very complicated, but these are all little ZeroMQ entities that do all the communication. It's actually very fast, very efficient, and doesn't bottleneck nearly as badly as Twisted used to.

This is a log plot. This is not a linear plot. On that log plot, what you're looking at is task latency — how many tasks can we send back and forth trying to saturate things out? With Twisted, we more or less saturate at roughly somewhere between 10 and 50 tasks per second empty. That's the frequency, which means you can't do anything that's high-frequency with the old implementation. Your tasks have to be chunky, because the latency cost of sending a task is pretty substantial. Worse than that, your throughput drops as the number of tasks sent gets higher.

It is important to note that most of these problems are not problems with Twisted. They're problems with our implementation with Twisted. Very, very important disclaimer. Twisted can actually, if you write good Twisted code — and if you actually could write Twisted code that's relatively similar to ZeroMQ code — it would actually be pretty close. But our understanding is that Twisted is extremely hard to write well.

The main cause of the huge drop is that the way the client is written, it's always receiving the results at the Python level. So if you've got results coming in, submitting new work is going to be slowed down by handling the results. And with ZeroMQ you can not do that until you ask for them, even though they'll be in local memory. In ZeroMQ they won't have hit Python yet, so they can't get in your way. Because they basically get buffered by the ZeroMQ object that is holding them until you ask for them. And when you ask for them, they're here, because they got buffered on the C++ side of things. When you make a request for an object, the request doesn't go over there — if the message was already sent, it's already sitting there, and you just fetch it locally.

And throughput is also substantially better. This is an area where Min did a lot of work on ensuring that no unnecessary copies were made. Not only are we also a couple of orders of magnitude faster in terms of the number of arrays we can send back and forth, but furthermore we only start dropping when the size of the arrays gets very, very large, whereas Twisted starts dropping significantly earlier, and that's partly because of the amount of copying that Twisted does. With ZeroMQ, Min worked very hard to ensure that no unnecessary copies were made. ZeroMQ has this clever way where you can tell it not to make copies of data if you're very careful about respecting the assumptions.

### DAG-Based Task Scheduling

So this thing exposes multiple usage patterns. You can have a direct interface where you can control directly where things get run, but you also have a load-balanced task interface. ### Closing: Open Tools for Open Science

So to wrap up, because I've already well overstayed my welcome, I think that in this audience I don't need to belabor the point that there's two sides to this. On the technical side, we have a very, very good tool for the kinds of problems that we were trying to do in today's scientific computing contexts, and Sage is an excellent example of using this to its full extent. But also social considerations — that these are tools that we are developing because we need them, because we want them.

I'm personally convinced that black boxes don't belong in research, at least in the critical path of research results. I don't care if you use a closed source tool to write your papers, whether you use Microsoft Word or LaTeX, that's up to you, what kind of thing you want to inflict upon yourself. But research results should not be hidden behind a black box that we're not allowed to verify and validate, and at least inspect and critique and understand. And I suspect we all agree on that point.

But also the last point is a point that is close to home for me. I come from Colombia, a country with very limited resources. And that's the same argument that India made — with the little money we have, should we spend that money in paying an annual rent to commercial software houses, or should we invest that money in developing tools where our students learn as part of the process of developing them, they become active participants of the scientific process, and they can further share what they're doing after the fact? And I think they made a very wise decision in that regard.

But there's still a lot to do, both on the Sage side of things and on the sort of numerical scientific side of things. And so I hope that those of you who are new to this at this workshop will go back to your home institutions and continue contributing, on whatever front it is that you end up contributing. So thanks a lot, and sorry for the very long talk. If there are any questions, if not, we can break out for other work, and I'm here until tomorrow night, so I'm more than happy over lunch, over coffee, at any point to chat with anyone, because I know it's been a long talk.
