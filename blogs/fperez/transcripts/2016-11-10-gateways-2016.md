### On the Underlying Protocol Architecture

Yes, Nancy will stop me. Do you mean web sockets? Yeah. This is all web sockets. Yeah. So the way the protocol is structured, the low level protocol between kernels that speak to kernels is JSON data traveling over 0MQ connections. And there's a very specific connection pattern that we present so that you can have both direct execution and listeners on broadcast. So we have both request reply and pop sub sockets for the different kinds of data.

But that 0MQ traffic is translated by a proxy into web sockets. And the actual packets basically go unmodified because there's pretty much a one-to-one mapping between the patterns that web sockets expose and what 0MQ exposes. And that's what the role of the kernel... if you want a notebook server, the notebook server does that. But if you don't want notebooks, the kernel gateway is basically the low level abstraction that does that proxying between JSON over 0MQ and web sockets. And the reason we chose that is because it allows pretty much any programming language that has a JSON encoding library, that these days is just about anything, and 0MQ bindings can do it. And it turns out 0MQ is a very, very compact C++ library with almost no dependencies. And so lots and lots of languages have 0MQ bindings.
