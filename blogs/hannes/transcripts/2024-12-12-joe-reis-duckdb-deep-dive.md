One thing you were talking about when we were having lunch. Somebody asked about memory versus DuckDB. Can you walk people through how to think about that? Yeah, people think there's a misconception that DuckDB is an in-memory database. It is not. We can use memory. We like memory. It's nice. It's fast. But we're not limited by it. With Lawrence, my PhD student, he's been working on DuckDB's larger than memory capabilities. We've always had this thing that your input data could be bigger than memory because we read it in a streaming way. If you point DuckDB at a Parquet file, it will not first copy the Parquet file into RAM and then do things with it. That would be dumb. Instead, it starts on the first row group, second row group, and so on.

However, there are operators in relational analysis, so-called blocking operators, like join, sort, aggregate, some window functions, top-N, that may have to materialize a large amount of the input. So if you're loading a terabyte of data, we might read it in a streaming fashion. But if you aggregate on the unique key, we will have to put it into our hash table for the aggregation. The semantics of SQL dictate this. And it's pretty common in analytical systems to actually fail or do something very slow in this case.

But in DuckDB, we built something called graceful degradation. When we reach the memory limit, we start using the disk. And that's only really possible because we have these crazy SSDs now. We can offload to disk at six gigabytes per second. We can read it back at six gigabytes per second. So we don't get slowed down too much. It's really improved compared to the spinning rust kind of thing. So we can for things like aggregations offload part of the intermediate result to disk and basically resume. And you won't feel a performance cliff there because it will still use as much memory as you allow us to use. But it will also be able to use the disk.

This gets even more interesting if you have a join because you can have multiple joins in the same query. Now you have multiple operators running at the same time, fighting for memory at the same time. And you have to think about a fair allocation strategy between those operators. We have a paper that Lawrence wrote, currently under revision at VLDB, which is a large database conference, that describes how we've implemented this strategy in DuckDB. How does ACID work in that environment? In the situation where you have a small amount of memory, how do you make sure that the transactions are committed or rolled back? This is independent of query processing. When a query starts running, MVCC, the Multi-Version Concurrency Control, will read a specific version of the table. That's going to be consistent during the runtime of the query. That has nothing to do with memory. It's just we read this version.

What is more interesting is how we deal with changes that are made within a transaction that are bigger than RAM. And that we're also able to offload. The write-ahead log, where changes go before transaction commit, will of course be written at the end of the transaction. But we do speculatively write big changes already to the database file. Let's say you have a table, you're inserting a terabyte CSV file in a transaction and then committing. Traditionally, what would happen is you're writing that terabyte file in your write-ahead log because you need to be transactionally safe. Then at checkpoint, you read it again and actually write it to your persistent table, at which point you truncate your write-ahead log. Now you've basically written this terabyte file twice to disk and read it twice. That's both read and write amplification.

So in DuckDB, what Mark has built is speculative writing of large changes. We will already write in the database file, but only at transaction commit into the WAL we only write references to those blocks. And only at checkpoint we essentially mark these blocks as being used in the main header. So in the default case where it works out, we write it once to the database file and that's it. Some metadata has to be updated later. And in the fail case where we abort, we'll have written some blocks to the database file that we then mark as empty. The worst thing that happened is we made the file a terabyte bigger. But that space will be reusable later on. So we've thought about this. The default case is going to be fast. The worst case is we use some disk space. It's a good trade-off from our perspective. These are the sorts of things we do in DuckDB to make this work. Nobody has been there before. Nobody has tried to write a terabyte to a WAL or not to write a terabyte to a WAL.
